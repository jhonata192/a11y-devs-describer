1. MELHORIAS TECNICAS (Bugs / Code Smells)
----------------------------------------------------------------------

1.1. config.yaml nao utilizado
     - O arquivo config.yaml com configuracao LiteLLM existe na raiz mas o
       codigo nunca o le. Remover ou integrar com o settings.py.

1.2. Ollama usa API Key obrigatoria desnecessariamente
     - core/ai/ollama.py exige OLLAMA_API_KEY para funcionar, mas servidores
       Ollama locais geralmente nao usam autenticacao. Tornar a chave opcional.

1.3. Cache version hardcoded
     - Em core/orchestrator.py:15, CACHE_VERSION = "ollama-v1" esta fixo mesmo
       quando o cliente configurado e OpenRouter. Deveria ser dinamico baseado
       em settings.ai_client.

1.4. pandoc_ast_builder.py truncado se texto muito longo
     - A funcao _meta_inlines divide o texto em chunks de 500 caracteres sem
       considerar palavras, podendo quebrar palavras no meio.

1.5. Tesseract nunca usado
     - O schema SQLite tem tabelas ocr_raw/revised/translated, e ha dependencia
       pytesseract no requirements.txt, mas o fluxo de OCR classico nunca e
       executado (o codigo usa IA de visao diretamente). Remover ou
       implementar.

1.6. Imports dentro de funcoes em start.py
     - Em interfaces/telegram/handlers/start.py, imports como
       "from interfaces.telegram.handlers.document import user_modes" sao
       feitos dentro do corpo das funcoes ao inves de no topo do arquivo.
       Funciona, mas viola boas praticas e confunde leitura.

1.7. Variavel _last_text em StatusTracker sem uso real
     - Em interfaces/telegram/adapters/status_tracker.py:59, _last_text e
       atribuida mas nunca lida em lugar nenhum.

1.8. UnifiedQueue faz polling ativo asyncio.sleep(0.5)
     - Em core/services/queue_service.py:63, o worker fica em loop com sleep,
       ao inves de usar asyncio.Condition ou asyncio.Queue para acordar quando
       um item chega.

5. PERFORMANCE
----------------------------------------------------------------------

5.1. Exportacoes paralelas no web app
     - Em interfaces/web/app.py, export_txt, export_docx, export_pdf rodam em
       ThreadPoolExecutor mas de forma sequencial. Poderiam rodar em paralelo
       com asyncio.gather.

5.2. Compressao adaptativa de imagem
     - Ajustar qualidade JPG baseado no tamanho da imagem. Imagens grandes
       (>5MB) podem usar qualidade 70 ao inves de 85.

5.3. Streaming de audio
     - Nao precisa esperar o MP3 completo para comecar o download. Poderia
       gerar e enviar chunk a chunk.

5.4. Lazy loading de modulos
     - Varios imports sao feitos no topo mas usados apenas em caminhos
       especificos (ex: fitz, PIL, cv2, numpy). Carregar sob demanda
       poderia reduzir tempo de inicializacao.

6. SEGURANCA
----------------------------------------------------------------------

6.1. Sanitizacao de filename no upload web
     - Em interfaces/web/app.py:174, document_file.filename e usado
       diretamente como path. Passivel de path traversal.

6.2. Rate limiting no web app
     - Atualmente nao existe limite de requisicoes. Adicionar
     slowapi/limiter para evitar abuso.

6.3. Validacao de token CSRF
     - Os formularios web nao possuem protecao CSRF. FastAPI nao faz isso
       automaticamente.

6.4. Timeout por requisicao
     - O request_timeout de 3600 segundos (1 hora) pode ser muito alto.
     Implementar timeout progressivo ou por etapa.
